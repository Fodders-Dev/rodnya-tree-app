# Branches / Circles UX rework — design plan

## Что не так сейчас

Список из 17 веток в audience picker — это правда «тихий ужас», как
описал пользователь. Даже после grouped-rework (`453aca7`) у этого
подхода остаётся фундаментальная проблема: **пользователь думает не
названиями веток, а лицами** — «папа, мама, я, сестра, её муж,
племянник, моя девушка». Перевести это в «Ветка: Кузнецов Артем
Андреевич / Предки: Кузнецова Наталья Геннадьевна» — лишний
когнитивный шаг.

## Что нравится сейчас

- Активная карточка в дереве подсвечивает связи (как обводка +
  цветные ребра).
- Полная ручная карта дерева — главное место, где живёт ментальная
  модель пользователя.

## Цель

Сделать выбор аудитории графическим: пользователь касается людей на
дереве, а не строк в списке. Ветки и предки — это **подсказки**, а не
обязательный путь. Сохраняем «Всё дерево» / «Избранные» как один-тап
варианты для 80% случаев.

## Концепция: «Tree-as-picker»

Когда пользователь хочет уточнить аудиторию, он видит не отдельный
modal со списком, а **то же самое дерево, что в /tree**, в режиме
выбора:

1. Карточки слегка приглушены, активные — подсвечены акцентом.
2. Tap по карточке → toggle (включить/выключить).
3. Long-press по карточке → quick-actions:
   - «Добавить ветку (этот человек + потомки)»
   - «Добавить линию предков»
   - «Только этот человек»
4. Сверху — счётчик-pill: «Видят 5 человек», тап раскрывает список.
5. Снизу — sticky-bar:
   - **Всё дерево** (default)
   - **Избранные** (если сохранены)
   - **Моя семья** (новый smart-set, см. ниже)
   - **Кастом** (что выбрано в данный момент)
6. Готово/Отмена в правом-верхнем углу.

## Smart-сеты (built-in shortcuts)

Чтобы избавить пользователя от 80% ручного выбора:

### «Моя семья» — auto-circle первого порядка
Алгоритм: anchor = текущий пользователь, включаем:
- родителей
- партнёра
- братьев/сестёр + их партнёров
- детей + их партнёров (если есть)

Это даёт ровно тот набор «папа, мама, я, сестра, её муж, племянник,
моя девушка», который пользователь описал. Backend уже знает relations,
backend-side можем считать на лету.

### «Близкие» — расширенный сет
- Всё из «Моя семья»
- Бабушки/дедушки
- Дяди/тёти
- Двоюродные братья/сестры

### «Поколение»
- Только сверстники (одно поколение)
- Только младше / старше

Эти сеты — не статичные circles, а живые predicates над relations.
Backend computes их каждый раз, frontend кэширует на сессию.

## Граф-выбор: техническая концепция

**Frontend:**

1. `BranchPickerScreen` или `BranchPickerSheet` — переиспользует
   существующий `TreeView` widget с props:
   - `mode: TreeViewMode.picker` (новый enum)
   - `selectedPersonIds: Set<String>` контролируется host'ом
   - `onPersonTap` / `onPersonLongPress` callback'ы

2. В `TreeView`-picker-mode:
   - Карточки с `isSelected=false` отрисовываются с opacity 0.55
   - Карточки с `isSelected=true` — полная opacity + accent border
   - Связи (ребра) — серые везде, accent green между selected
   - При long-press показывается quick-action `BottomSheet`:
     ```
     • Только этот человек
     • С потомками
     • С линией предков
     • Все в этой семье (партнёр + дети)
     • Отменить выбор
     ```

3. Top-bar pill «Выбрано: N человек» — тап раскрывает sheet со списком
   людей и кнопками «Очистить» / «Выбрать всех».

4. Bottom sticky-bar с smart-сетами и Готово.

**Backend:**

Новые endpoints (могут жить под `/v1/audience-presets/...`):
- `GET /v1/trees/:treeId/audience-presets` — возвращает рассчитанные
  smart-сеты для текущего пользователя (anchor):
  ```json
  [
    {"key": "all", "label": "Всё дерево", "personIds": [...18 ids...]},
    {"key": "favorites", "label": "Избранные", "personIds": [...]},
    {"key": "core_family", "label": "Моя семья", "personIds": [...7 ids...]},
    {"key": "close", "label": "Близкие", "personIds": [...]},
    {"key": "generation_self", "label": "Моё поколение", "personIds": [...]}
  ]
  ```

`createPost` уже принимает `anchorPersonIds: List<String>` — никаких
изменений не нужно. Просто frontend стал отправлять список personIds,
выбранных в графе.

## Маппинг к существующей audience модели

Сейчас в коде:
- `scopeType: TreeContentScopeType.wholeTree | branches`
- `circleId` (для системных кругов вроде «Избранные»)
- `anchorPersonIds: List<String>` (для веток)

После rework:
- **Quick-pick «Всё дерево»** → `scopeType: wholeTree, circleId:
  null, anchorPersonIds: []` (как сейчас).
- **Quick-pick «Избранные»** → `scopeType: wholeTree, circleId:
  favoritesId, anchorPersonIds: []`.
- **Smart-сет «Моя семья» / «Близкие»** → `scopeType: branches,
  circleId: null, anchorPersonIds: <вычисленные ids>`.
- **Кастом-граф-выбор** → `scopeType: branches, circleId: null,
  anchorPersonIds: <выбранные ids>`.

То есть структура storage-уровня не меняется. Меняется только UI.

## Этапы внедрения

### Phase 1 — smart-сеты (один спринт)
Самый дешёвый wins:

- Backend: `GET /v1/trees/:treeId/audience-presets` возвращает
  `core_family` и `close` сеты. Логика: для anchor=user.personId
  обходим relations.
- Frontend: добавить «Моя семья» и «Близкие» в существующий
  `AudiencePicker` как тайлы первого порядка (наряду с «Всё
  дерево» / «Избранные»). Скрытьавто-сборку «Ветви потомков /
  Линии предков» под кнопку «Подробнее» — теперь нужны редко.

После этого 80% случаев решаются 2-3 тайлами без захода в граф.

### Phase 2 — графический picker (отдельный спринт)
- Новый `BranchPickerScreen` route
- TreeView в picker-mode (новый enum + props)
- Long-press quick-actions
- Top-bar counter, bottom sticky-bar с smart-сетами

Вход: тап на «Кастом» в Phase 1 picker'е → push picker screen.

### Phase 3 — saved custom circles (опционально)
- Если пользователь часто использует один и тот же кастом-набор —
  кнопка «Сохранить как круг» в bottom-bar picker'а.
- Pamento в `circles` со `kind: custom`. Picker подхватит как
  очередной тайл.

## Открытые вопросы

1. **Anchor — это `userId` или `personId`?** В дереве может быть
   несколько person-cards привязанных к одному userId (через
   identityClaims). Нужно решить, считаем ли smart-сет от primary
   personId или от всех связанных.

2. **Cross-tree smart-сеты.** Сейчас audience scoped per tree. Для
   мульти-tree аккаунтов — отдельный сет на каждое дерево? Считаем,
   что да.

3. **Сохранение состояния picker'а.** Если пользователь потратил
   время на ручной граф-выбор и закрыл picker — потерялось? Скорее
   всего да в v1. В v2 можно сохранять draft.

4. **Stories vs posts.** Pic'ер один на оба, или разные? Думаю —
   один с разной конфигурацией smart-сетов (для stories отдельный
   близкий круг может быть полезен).

5. **Performance.** На больших деревьях (100+ людей) graph-picker
   может тормозить — нужен виртуализированный rendering или
   кластеризация. Существующий tree-view уже это решает.

## Что я предлагаю сделать первым

**Phase 1 (один-два коммита):**
- Backend: `audience-presets` endpoint с `core_family` и `close`.
- Frontend: добавить smart-tile'ы в audience picker.
- Auto-сборку веток под expandable-секцию (как сейчас Linии
  предков), но collapsed-by-default.

Это даёт мгновенное облегчение для 80% случаев и не требует пилить
новый picker screen целиком. Phase 2 — отдельная волна, более
амбициозная и UI-heavy.

# Большой prompt для Codex в облаке

```text
Ты работаешь с репозиторием «Родня» на GitHub.

Сначала обязательно прочитай и соблюдай:
- AGENTS.md
- CODEX.md
- Codex_rules.md
- PROMPT.md

Контекст продукта:
- «Родня» — это семейное дерево + приватная семейная социальная сеть для родственников.
- Главный фокус: стабильный Android + web MVP, пригодный для ежедневного использования.
- Ближайший релизный приоритет: RuStore-first.
- Firebase-hosted и иные внешне-зависимые cloud-paths считаем legacy, если они мешают доступности в России.
- Целевое направление backend: custom API + PostgreSQL + object storage + свой realtime/push-compatible path.

Критичные продуктовые ограничения:
- НЕ менять текущий Android applicationId, потому что он нужен для апдейта существующей карточки в RuStore.
- НЕ трогать store identifiers, signing, release-specific IDs и чувствительные секреты.
- НЕ делать deploy, не работать с сервером, не использовать локальные секреты.
- НЕ придумывать новую архитектуру рядом со старой. Улучшай существующую.
- НЕ тащи большие бесполезные текстовые блоки в UI. Приоритет — плотный, минималистичный, полезный интерфейс.
- Все пользовательские тексты на русском должны быть естественными, короткими и нормальными для живого продукта.

Рабочая стратегия:
1. Сначала изучи текущий код по затронутым путям.
2. Определи реальный scope задачи.
3. Делай минимальные, обратимые изменения.
4. Если задача затрагивает бизнес-логику, выноси её в services/repositories/providers, а не раздувай screen.
5. Если задача меняет интерфейс, убирай лишнюю “текстовую массу” и повышай полезную плотность.
6. Если задача меняет backend contract, обнови интерфейсы, адаптеры и тесты вместе.
7. Если задача затрагивает web UI, проверяй compile path и делай короткий smoke.

Стандарт качества для этого проекта:
- приложение должно быть не просто “с фичами”, а реально удобным для ежедневного использования;
- дерево должно оставаться центральным объектом, а не утопать в helper-панелях;
- посты, события, профиль, родственники и чаты должны ощущаться как цельный продукт;
- UI должен становиться минималистичнее и полезнее, а не тяжелее;
- ошибки backend/client sync надо чинить через реальный контракт и явную синхронизацию состояния, а не костылями;
- если видишь product issue, а не только code issue, исправляй или явно отмечай риск.

Что особенно важно в Родне:
- family tree mode и friends tree mode — оба first-class сценарии;
- родственники и семейные события — ядро ценности;
- совместное редактирование дерева должно быть удобным и понятным;
- медиа на Android/web должно открываться и выглядеть стабильно;
- главная, дерево, профиль и карточки родственников не должны быть забиты длинными описаниями “ради текста”.

Ожидаемый стиль изменений:
- не делай широких хаотичных рефакторингов;
- не меняй unrelated файлы;
- не оставляй полумёртвые промежуточные слои;
- не ломай совместимость, если можно мигрировать мягко;
- если есть legacy aliases, сохраняй их, пока не мешают.

Что считать definition of done:
- код реально изменён;
- поведение или UI реально улучшены;
- релевантные тесты добавлены или обновлены;
- `dart format` выполнен на изменённых Dart-файлах;
- `flutter analyze` должен проходить;
- релевантные `flutter test` должны проходить;
- если менялся web path, нужен короткий smoke;
- в отчёте должны быть: что изменено, что проверено, что ещё рискованно.

Формат работы:
- не останавливайся на анализе, если можно безопасно реализовать задачу;
- не задавай лишних вопросов, если разумное инженерное допущение очевидно;
- если scope большой, сначала составь краткий план выполнения, а потом иди по нему;
- если видишь более правильный минимальный путь к цели, выбирай его и объясни это в финале кратко.

Формат финального ответа:
- коротко перечисли, что изменил;
- перечисли проверки;
- отдельно назови оставшиеся риски или незакрытые хвосты;
- не пиши длинную воду.

Теперь сама задача:

<ВСТАВЬ СЮДА ОДИН УЗКИЙ CLOUD-SCOPE>

Примеры хороших cloud-scope для этого репозитория:
- rebrand UI/docs from Rodnya to Родня, without changing Android applicationId
- make likes server-driven across post service, widget state, and tests
- expand EventService with birthdays, anniversaries, memorials, RF holidays, orthodox holidays, and tests
- declutter HomeScreen and remove permanent greeting hero while preserving key actions
- add stories backend/service layer for 24h stories without release/deploy work
- add tree history API and append-only change records with tests

Примеры плохих задач для cloud:
- зайди на production сервер и выкати backend
- проверь на моём Android телефоне по USB
- залей билд в RuStore и прожми permissions
- используй локальные токены, секреты или консоли

Если в процессе видишь явный blocker:
- назови его конкретно;
- не раздувай scope;
- не трогай лишнее;
- всё равно доведи максимум безопасной части задачи до конца.
```

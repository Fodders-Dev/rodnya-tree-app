## Reported bugs (Артёмов список)

### 1. Audio one-way (Samsung A50, Huawei MatePad 11.5s)

**File:line**: `lib/services/call_coordinator_service.dart:913-920` (room.connect +
setMicrophoneEnabled sequence); `lib/screens/call_screen.dart:164-206` (ringer
AudioContext); `android/app/src/main/AndroidManifest.xml:6-28` (missing
permissions).

**Root cause analysis**: набор взаимодействующих факторов, в порядке от наиболее
до наименее вероятного.

1. **MISSING `FOREGROUND_SERVICE_MICROPHONE` permission (Android 14+).** Манифест
   декларирует только `RECORD_AUDIO` + `MANAGE_OWN_CALLS`. `targetSdk = 35`
   (`android/app/build.gradle:104`), значит на Android 14+ микрофон не может
   captureться из foreground service без специального permission. У Rodnya нет
   foreground service вообще для активного звонка — `RodnyaConnectionService`
   объявлен только для входящего telecom UI, не для media. Когда экран
   блокируется (или Telecom self-managed connection переключает focus), Android
   silently revokes mic capture у backgrounded процесса. Самsung A50 / Huawei
   MatePad — оба запущены на Android 13+ (вероятно 13–14). На Huawei специально:
   EMUI / HarmonyOS агрессивно background-killит mic streams без foreground
   service notification.

2. **`setMicrophoneEnabled` errors swallowed silently.** В `_ensureConnected`
   (line 919) `await room.localParticipant?.setMicrophoneEnabled(true)` — если
   операция throws (например, AudioRecord init failed на Huawei), исключение
   ловится единственным `try` блоком вокруг всего `room.connect`, и
   `_looksLikeMediaPermissionIssue(error)` ставит `_hasMediaPermissionIssue`.
   Но `_microphoneEnabled = true` уже выставлен на line 917 ДО await — UI
   показывает «mic on» хотя трек не опубликован. Проверка успеха publish
   отсутствует — нет inspection `localParticipant.audioTrackPublications` после
   await.

3. **AudioContext race с LiveKit AudioManager.** Ringback player (line 175-197)
   ставит `AndroidAudioMode.inCommunication` + `isSpeakerphoneOn: true` для
   outgoing call. Это меняет глобальный `AudioManager.mode` ДО того как LiveKit
   подключится. LiveKit WebRTC при `room.connect` тоже хочет управлять
   AudioManager. В момент когда ringer останавливается (`_stopRinger()` в
   `_handleCoordinatorChanged` line 295) audioplayers вызывает internal
   `setMode(MODE_NORMAL)`, что может выбить LiveKit из VoIP audio mode. На
   Samsung A50 это especially проблемный путь — One UI Audio HAL очень
   чувствительный к mode flips.

4. **`Permission.microphone.request()` returns true на «previously granted»**,
   но не верифицирует что AudioRecord can actually open the device. На Huawei
   при первом cold start permission grant'ы могут вернуть `isGranted` ещё до
   того как HAL allow'ит фактический capture. Нет `microphoneStatus.isPermanentlyDenied`
   check, нет retry-on-failed-capture pattern.

**Fix complexity**:
- (1) MEDIUM — добавить `FOREGROUND_SERVICE` + `FOREGROUND_SERVICE_MICROPHONE`
  permissions + завести foreground service `RodnyaCallForegroundService` который
  стартует при `_applyCall(state=active)` и стопится при terminal/disconnect.
  Нужна manifest change + new Kotlin service + Flutter MethodChannel binding.
- (2) TRIVIAL — wrap `setMicrophoneEnabled` в свой try/catch, на failure expose
  `microphonePublishFailed` flag через notifier, не trust `_microphoneEnabled` без
  верификации `audioTrackPublications.any((p) => p.source == TrackSource.microphone && !p.muted)`.
- (3) MEDIUM — defer ringer stop до ПОСЛЕ LiveKit подтвердил audio mode acquire,
  или сделать ringback playback через LiveKit AudioContext (не audioplayers).
  Альтернатива: stop ringer ДО `_applyCall` apply, через explicit hook.
- (4) MEDIUM — добавить smoke-test через actual AudioRecord.startRecording() с
  short read() перед декларацией permissions OK.

**Test verifiability**: device-only (нужны Samsung A50 + Huawei). Можно code-only
покрыть unit-tests для пункта (2) — assert что publication failure exposed
properly через notifier. Также code-only возможна regression test на проверку
manifest permissions в integration test.

**Verdict**: 95% pointer на (1) — это типичная регрессия Android 14 + WebRTC
apps без foreground service. Артём reported симптом «собеседник Артёма не
слышит» — это исходящий audio path, который полностью consistent с silent mic
capture revocation в background.

### 2. Self-preview missing на audio→video toggle

**File:line**: `lib/screens/call_screen.dart:84` (`_isVideoCall` getter);
`lib/screens/call_screen.dart:467-493` (`_localVideoTrack`);
`lib/screens/call_screen.dart:886-891` (PiP gate);
`lib/services/call_coordinator_service.dart:456-478` (`toggleCamera`);
`lib/services/call_coordinator_service.dart:673-675` (mediaMode-tied camera reset).

**Root cause analysis**: коллапс нескольких logical paths.

1. **`_isVideoCall` is server-derived, never updated by local toggle.** Getter
   возвращает `_resolvedCall.mediaMode == CallMediaMode.video`. Когда пользователь
   в audio call жмёт камеру, `_toggleCamera` вызывает
   `coordinator.toggleCamera()` → `setCameraEnabled(true)` (line 470). Сервер
   об этом не уведомляется — call record остаётся `mediaMode: audio`. Значит
   `_isVideoCall` остаётся `false`. Это влияет на:
   - chrome auto-hide (line 122): `_scheduleVideoChromeHide` skip'ает audio
   - tap-to-toggle-chrome (line 641): disabled для audio
   - main stage routing: `_pipShowsLocal || localVideoTrack == null` → since
     `_pipShowsLocal=true` by default, falls into `_CallStage` (line 648) which
     renders ONLY the remote feed or avatar fallback. Local track ВООБЩЕ не
     попадает на main stage в audio→video upgrade path.

2. **PiP self-preview ДОЛЖЕН отрисоваться** (gate is `localVideoTrack != null`,
   line 886), но 120×180dp tile в bottom-right at `_pipShowsLocal=true` =
   правильно показывает local feed. Тем не менее, симптом «себя не вижу» имеет
   два альтернативных корня:
   - **Race condition**: `_toggleCamera` делает `setState(() {})` в `finally`
     блок line 388-390 ДО завершения coordinator notification, и publication
     может ещё не существовать. Coordinator->`notifyListeners()` на line 477
     приходит позже, но рекомпозиция `_localVideoTrack` getter может прийти
     ДО реального добавления publication в `videoTrackPublications` (LiveKit
     emits events asynchronously). Нет ParticipantEvent specifically на
     `TrackPublishedEvent` — coordinator слушает только generic `RoomEvent`
     (line 903) и `ParticipantEvent` (line 900) которые catch-all-ят. Этого
     должно быть достаточно, но на медленных устройствах (Huawei) задержка
     может выглядеть как «preview не появился».
   - **`_applyCall` resets `_cameraEnabled`**: line 673-675 — если приходит
     server snapshot с `mediaMode.isVideo == false` (т.е. для audio call) ПОСЛЕ
     toggleCamera, `_cameraEnabled = false` принудительно. Это вероятно
     произойдёт в `_refreshActiveCall` poll (line 738 interval=2s) который
     запрашивает `getCall` и applies snapshot. Через 2 секунды после
     audio→video toggle камера unmount'нится. **High-confidence regression
     trigger**.

3. **`_resolvedCall.mediaMode` доступ — UI hardcodes** «Аудиозвонок» в status
   label (line 529) даже когда камера активна. Это subtle UX bug: user видит
   «Аудиозвонок» but включил камеру — путаница.

**Fix complexity**:
- (1) HARD — model'ить «local upgrade to video» как client-only state, либо
  добавить server-side endpoint `POST /v1/calls/:id/upgrade-media` который
  меняет mediaMode на сервере и broadcasts. Last preferred — единственный
  способ не получить spurious downgrade на следующем `_refreshActiveCall`. До
  тех пор — local override flag `_localCameraUpgrade` который должен
  ignored'ить server mediaMode для UI purposes.
- (2.a) TRIVIAL — добавить explicit `TrackPublishedEvent` handler в room
  listener (line 854) который форсит `notifyListeners()`.
- (2.b) MEDIUM — игнорировать `_cameraEnabled` reset когда `_localCameraUpgrade`
  set. Или: не сбрасывать camera state в `_applyCall` если call.id не
  changed, только если новый call entirely.

**Test verifiability**: code-only для (2.b) — widget test: audio call active,
вызывает coordinator.toggleCamera(), потом второй `activateCall` с tom же
snapshot, expect cameraEnabled === true (currently fails).

### 3. Audio→video toggle unreliable

**File:line**: `lib/services/call_coordinator_service.dart:456-478` (`toggleCamera`);
`lib/services/call_coordinator_service.dart:670-675` (camera state reset on apply).

**Root cause analysis**: same root cause as Bug #2 пункт 2.b — гонка между
`toggleCamera` и `_refreshActiveCall` 2s polling (line 738). Дополнительные
факторы:

1. **No idempotency check на `toggleCamera`**. Если пользователь tap'ает много
   раз быстро, каждый tap инвертирует `_cameraEnabled` и await'ит setCameraEnabled
   (медленно — захватывает камеру). Можно получить интерлив `setCameraEnabled(true)` →
   `setCameraEnabled(false)` → ... с непредсказуемым итоговым state. Нет
   `_isTogglingCamera` guard (есть `_isSwitchingCamera` но он только для front/back swap).

2. **`CameraCaptureOptions(cameraPosition: _cameraPosition)` на каждом
   toggle** — каждый раз заново allocate'ит camera capture pipeline. На Huawei
   это slow, может зависнуть на 2-3 секунды.

3. **Permission check НЕ перепроверяется** — `toggleCamera` запрашивает камеру
   через `setCameraEnabled` напрямую без `Permission.camera.request()`. Если
   permission revoked после connect (рейс OS), throw уходит в caller (line
   379-387) и пользователь видит snackbar. Но `_cameraEnabled = true` уже
   выставлен — desync state.

**Fix complexity**:
- (1) TRIVIAL — добавить bool `_isTogglingCamera`, return early if already true.
- (2) MEDIUM — кэшировать `CameraCaptureOptions` instance, переиспользовать.
- (3) MEDIUM — пре-проверить camera permission через permission_handler перед
  setCameraEnabled, или surface failure properly.

**Test verifiability**: code-only для (1) — fire-and-await 5 toggles in a row,
expect только final cameraEnabled state.

### 4. Video PiP drag broken

**File:line**: `lib/screens/call_screen.dart:921-1008` (`_buildDraggablePip`).

**Root cause analysis**:

1. **Pan recognition disambiguation работает корректно по коду**, но есть несколько
   sharp edges:
   - `dragStartBehavior: DragStartBehavior.down` (line 932) means drag начинается
     на pointer-down БЕЗ slop — хорошо для responsiveness. Однако комбинация с
     manual 6.0-dp threshold на line 950 создаёт inconsistency: pan recognizer
     уже claim'ил pointer, но callback гейтит первые `< 6.0` dp. Если родительский
     `GestureDetector` (например `Scaffold.body` Stack) тоже захочет pointer
     (например, для tap-on-stage chrome toggle, line 639), может произойти
     ambiguous winner. The outer GestureDetector at line 639 is `Positioned.fill`
     с `onTap` defined только когда `_isVideoCall && active`. Если оба recognizer
     конкурируют, родительский может claim первым (особенно потому что у PiP
     `Positioned` без явной z-order priority).

2. **`_pipDragStartOffset` инициализируется в `onPanStart`** (line 941), но
   `dragStartBehavior: DragStartBehavior.down` означает что **`onPanStart`
   может НЕ срабатывать** (down behaviour не запускает start — drag start
   fires только когда есть meaningful delta). Это значит `_pipDragStartOffset`
   может быть `null` на первом `onPanUpdate`. Fallback на line 951
   `(_pipDragStartOffset ?? offset)` спасает, но `_pipDragAccumulated`
   аккумулируется правильно.

3. **`_LocalFullStage` (line 1015-1032) перекрывает PiP** когда `_pipShowsLocal == false`:
   PiP rendering at z-top of stack но `_LocalFullStage` rendered как
   `Positioned.fill` ВНУТРИ outer `GestureDetector`. Layer order в Stack:
   `Positioned.fill` (stage) → `SafeArea` (chrome) → PiP (Positioned). PiP
   ДОЛЖЕН быть на top.

4. **`HitTestBehavior.opaque`** на PiP `GestureDetector` (line 927) — good,
   stops propagation. **Тем не менее**, parent `GestureDetector` line 639-660 с
   `HitTestBehavior.opaque` тоже не должен receive pointer'ы, попадающие в PiP
   bounds. Но если PiP за пределами viewport (due to clamp bug) или z-order
   wrong, может оказаться что parent claim'ает первым.

5. **Real bug found**: `_pipOffset` clamping на line 954-960 использует
   `pad.top + 8.0` как min Y. Если у пользователя есть system gesture bar
   (Android 10+, especially Samsung A50 с навигацией), `pad.top` может быть
   маленьким и PiP застрянет высоко. Min Y должен учитывать chrome height (header
   panel ~120dp при visible chrome).

6. **PiP не имеет `onPanDown`** который явно claim'ит gesture. На некоторых
   устройствах (особенно Huawei) Flutter может не заметить «down» phase
   правильно если pointer landing был at edge.

**Most likely cause** — пункт 1 (parent gesture conflict) + пункт 2 (start
behavior down + manual threshold disambiguation). User сообщает «окошко не
двигается», что consistent с parent claim'ом gesture.

**Fix complexity**:
- MEDIUM — переписать с использованием `Draggable`/`LongPressDraggable` или
  custom `RawGestureDetector` с явным `OneSequenceGestureRecognizer` priority.
  Альтернатива: wrap PiP в `Listener` для raw pointer events, manual hit-testing
  без participation в gesture arena.
- TRIVIAL — изменить `dragStartBehavior: DragStartBehavior.start`, убрать
  ручной 6-dp threshold, использовать `kTouchSlop` через `framework.gestures`.

**Test verifiability**: device-only — gesture arena поведение тяжело unit-test'ить.
Можно widget-test через `tester.timedDrag()` чтобы verify state mutation, но
real device-touch behavior может отличаться.

## Discovered bugs (worker found)

### Bug A: Foreground microphone service отсутствует — Android 14+ silently revokes mic capture

**Severity**: CRITICAL
**File:line**: `android/app/src/main/AndroidManifest.xml:6-28`; no `RodnyaCallForegroundService` Kotlin class exists.
**Description**: `targetSdk = 35`, нет `FOREGROUND_SERVICE` permission, нет
`FOREGROUND_SERVICE_MICROPHONE` permission, нет foreground service запущенного
во время active call. На Android 14+ media capture в background — silent revocation
unless app holds an active foreground service with the matching type. Это root
cause for Bug #1 (audio one-way) и потенциально других «звук пропадает» симптомов.
**Root cause**: project никогда не имел call media foreground service. Telecom
self-managed connection (`RodnyaConnectionService`) — это `BIND_TELECOM_CONNECTION_SERVICE`,
не media capture service.
**Fix complexity**: MEDIUM. Create `RodnyaCallForegroundService extends Service`,
ServiceType `mediaPlayback|microphone|camera`, persistent notification. Start
в `_applyCall(active)`, stop в `_disposeRoom`. Manifest permissions:
`FOREGROUND_SERVICE`, `FOREGROUND_SERVICE_MICROPHONE`, `FOREGROUND_SERVICE_CAMERA`
(для видеозвонков), `FOREGROUND_SERVICE_PHONE_CALL` (для telephony semantics).

### Bug B: Missing `BLUETOOTH_CONNECT` permission — BT audio routing будет fail на Android 12+

**Severity**: HIGH
**File:line**: `android/app/src/main/AndroidManifest.xml`; `lib/services/audio_route_service.dart:229-235` (bluetooth route construction).
**Description**: `AudioRouteOption(type: AudioRouteType.bluetooth)` создаётся,
если LiveKit's `Hardware.instance.audioOutputs()` возвращает BT device. На
Android 12+ (S+) попытка `setAudioOutputDevice` для BT device требует
`BLUETOOTH_CONNECT` runtime permission. Manifest не объявляет ни
`BLUETOOTH_CONNECT`, ни `BLUETOOTH` (legacy). User'ы с BT гарнитурой не смогут
переключиться на неё mid-call.
**Root cause**: BT permissions просто упущены. Manifest перечисляет camera /
mic / network / contacts / vibrate / wake_lock / boot / notifications /
fullscreen / modify_audio / manage_calls. BT отсутствует целиком.
**Fix complexity**: TRIVIAL — добавить `<uses-permission android:name="android.permission.BLUETOOTH_CONNECT"/>`
+ runtime permission request hook в `AudioRouteService` перед selectRoute(BT route).

### Bug C: iOS — нет CallKit / PushKit integration, нет VoIP background mode

**Severity**: HIGH (для iOS roadmap; LOW если iOS не shipped в ближайшее)
**File:line**: `ios/Runner/Info.plist:1-54`.
**Description**: Info.plist декларирует только `NSMicrophoneUsageDescription`
+ `NSCameraUsageDescription`. Нет `UIBackgroundModes` с `voip` или `audio`,
нет CallKit framework integration, нет PushKit handler. На iOS звонки будут
работать ТОЛЬКО когда приложение в foreground. Backgrounded → push не разбудит
для звонка. Также: iPad'ам не появится native incoming call UI (Telecom-эквивалент).
**Root cause**: iOS call subsystem никогда не был implemented в native side.
Backend / Flutter код работает (потому что pure cross-platform), но Apple
требует native CallKit для proper VoIP behavior.
**Fix complexity**: HARD — full CallKit + PushKit native module (Swift), iOS
push certificates configuration, APNs delivery setup. Out-of-scope для этой
phase, но critical для App Store submission.

### Bug D: `_handleCoordinatorChanged` Navigator pop fires for ANY null currentCall — multiple CallScreens cascade

**Severity**: HIGH
**File:line**: `lib/screens/call_screen.dart:274-283`.
**Description**: `if (coordinatorCall == null) { ... Navigator.of(context).maybePop(); ... }`
без any callId comparison. Если несколько CallScreens когда-либо стекаются (e.g.

### Bug E: Ringer player не cancelled при mid-init coordinator clear — orphan AudioPlayer

**Severity**: MEDIUM
**File:line**: `lib/screens/call_screen.dart:148-206`.
**Description**: `_startRinger` creates AudioPlayer и assigns `_ringerPlayer = AudioPlayer()`
на line 156. Затем async `await player.setAudioContext(...)` (line 175) и т.д.
Если в этот момент `_stopRinger()` triggered (call accepted / cancelled), он
выставляет `_ringerActive = false`, отменяет timer, `_ringerPlayer = null` и
вызывает `player.stop()`. **НО** — async block в `unawaited(() async { ... })`
(line 164-206) уже захватил local `player` reference и продолжает выполнять
setAudioContext + setReleaseMode + setVolume + play. После `dispose()`'а в
stopRinger, эти operations могут throw на закрытом player. `.catchError((_) {})`
на dispose маскирует, но: после play'а будет беззвучно loop'аться dead player
если timing exactly wrong.
**Root cause**: race window между `_startRinger` async init и `_stopRinger` cancel.
**Fix complexity**: TRIVIAL — внутри async block проверять `if (!_ringerActive) return;`
между awaits, или use cancellation token pattern.

### Bug F: `_applyCallPreferences` enumerates devices ДО `_microphoneEnabled` published — preferred mic may apply pre-publication

**Severity**: MEDIUM
**File:line**: `lib/services/call_coordinator_service.dart:937-940`.
**Description**: Sequence: `setMicrophoneEnabled(true)` (line 919) → `_audioRouteService.attachRoom`
(line 938) → `refreshInputDevices` (line 939) → `_applyCallPreferences` (line 940).
`_applyCallPreferences` → `_applyPreferredMicrophone` → `room.setAudioInputDevice(device)`.
Если `setMicrophoneEnabled` НЕ завершил publication к моменту `setAudioInputDevice`,
второй вызов может либо noop'нуть либо republish с разными capture options.
LiveKit 2.7.0 supposed to handle gracefully, но flaky behaviour observed in
community issues (LiveKit Flutter SDK issues #X-Y).
**Root cause**: ordering — apply preferences ДО verification что initial publication
settled.
**Fix complexity**: TRIVIAL — `await Future.delayed(small)` или wait for
`LocalTrackPublishedEvent` before applying preferences.

### Bug G: `_handleRoomDisconnected` always triggers reconnect — нет идемпотентности

**Severity**: MEDIUM
**File:line**: `lib/services/call_coordinator_service.dart:955-973`.
**Description**: `RoomDisconnectedEvent` handler сразу `_isReconnectingRoom = true`
+ `await refreshCurrentCall()`. `refreshCurrentCall` → `_callService.getCall` →
`_applyCall(refreshedCall)`. Если `refreshedCall.state == active`, `_applyCall`
вызывает `_ensureConnected(call, reconnect: true)`. Гонка: если
`RoomDisconnectedEvent` пришёл за миллисекунды до user hangup, или backend
вернул old snapshot до server-side cleanup, можно получить infinite reconnect loop:
disconnect → refresh → applyCall(active) → ensureConnected → connect →
disconnect... Нет backoff, нет max retry counter.
**Root cause**: missing reconnect attempt counter / exponential backoff.
**Fix complexity**: MEDIUM — добавить `_reconnectAttempts` counter, max 3 за
30 секунд, после max — show «Не удалось переподключиться» + finish call.

### Bug H: `_localVideoTrack` getter creates rebuild loop через `notifyListeners` chain

**Severity**: LOW (performance, not functional)
**File:line**: `lib/services/call_coordinator_service.dart:900-905`; `lib/screens/call_screen.dart:467-493`.
**Description**: Coordinator слушает все `ParticipantEvent` + `RoomEvent` и

high event frequency (track stats updates fire ~every 1s), это становится
N×rebuilds. Малозначимо при 1 видео-tile, но в group call (deferred, но архитектура
готова) — O(participants × events) per second.
**Root cause**: catch-all event handler без debouncing / event-type filtering.
**Fix complexity**: MEDIUM — filter event types: rebuild только на TrackSubscribed,

### Bug I: Push payload `callerName` уязвим к caller spoofing — нет signature verification

**Severity**: MEDIUM (security)
**File:line**: `lib/services/call_coordinator_service.dart:1330-1353`.
**Description**: `_handlePushMessage` берёт `message.data['callerName']` напрямую

### Bug N: `CallScreen.dispose` не stop'ает ringer player synchronously — UI dismissed но звук продолжается

**Severity**: MEDIUM
**File:line**: `lib/screens/call_screen.dart:108-118`.

**Root cause**: fire-and-forget cleanup в dispose.
**Fix complexity**: TRIVIAL — `await player.stop()` в widget didChangeAppLifecycle

### Bug O: `setMicrophoneEnabled` failure не reflects в UI — toggle button показывает stale state

**Severity**: MEDIUM
**File:line**: `lib/services/call_coordinator_service.dart:444-454`;

## Architectural concerns

1. **Singleton `AudioPlayer` for ringer not lifecycle-managed beyond CallScreen.dispose**
   — `lib/screens/call_screen.dart:53-54`. Если call switched (rare but possible:
   incoming-during-active), ringer player может leak. См. Bug E + Bug N.

2. **`CallCoordinatorService` is a god-object** — 1469 LOC, manages: signaling, room
   lifecycle, audio routes, device enumeration, preferences, push handling, Telecom
   bridge, ringing recovery, active call recovery, app lifecycle, vibration. Single
   class with 30+ private fields. Refactor target — extract `CallRoomLifecycleService`,
   `CallRecoveryService`, `CallTelecomCoordinator` separately. **НЕ DO в Phase 2** —
   Артём explicit «не propose architectural rewrites». Note as long-term debt.

3. **Polling Recovery Timers (`_ringingRecoveryTimer` 5s, `_activeCallRecoveryTimer` 2s)
   redundant с realtime subscription** — `lib/services/call_coordinator_service.dart:707-805`.
   2-секундный poll на каждый active call — серьёзный battery + network drain,
   особенно если realtime WS работает. Должен disabled когда realtime healthy и
   только enable на disconnect. Currently always-on.

4. **`_handleCoordinatorChanged` cascade**: `_applyCall` → `notifyListeners` →
   CallScreen `_handleCoordinatorChanged` → `_syncRinger` + lifecycle SFX +
   joinedOnAnotherDevice snackbar + Navigator.pop. **Multiple side-effects in
   single notifier callback** — error in any propagates / breaks subsequent. Нет
   guard если CallScreen dispose'нут mid-callback.

5. **`_disposeRoom` is async fire-and-forget в нескольких местах** — e.g.
   `lib/services/call_coordinator_service.dart:596, 628, 686, 703, 1402-1414`.
   Если новый звонок accepted ДО завершения предыдущего `_disposeRoom`, можно
   получить два concurrent room.dispose() операции с overlapping listener cancellation.

6. **State mutations не atomic** — `_applyCall` modifies 10+ fields в нескольких
   `notifyListeners` calls. Если listener делает что-то async и читает state
   между notifies, может увидеть inconsistent intermediate. Например: line
   677 `notifyListeners()` после mutation but ДО room reconnect — listener
   читает `_currentCall.state == active` but `_room == null` (would still
   match `state==active && room==null` branch in `_statusLabel`).

7. **`_pushCallerNames` map никогда не cleared полностью** — `lib/services/call_coordinator_service.dart:152`.
   Entries removed только на `_applyCall(null)` и `_applyCall(terminal)`. Если
   user никогда не accept'ает push call (просто игнорирует), entry stays
   forever. Memory leak на long-running app.

8. **`_visibleCallScreenIds: Set<String>` accumulates if `setCallScreenVisible(false)`
   never called** — `lib/services/call_coordinator_service.dart:146`. Memory leak
   if CallScreen GC'ed without dispose firing (rare but possible on hot-reload
   or process crash). Bound the set or use `WeakReference`-style approach.

9. **No CallSession token expiry handling** — `lib/services/call_coordinator_service.dart:80-90`.
   `livekit-service.js` создаёт token with `ttl: "2h"`. Если call длится > 2 часа,
   token expires mid-call → reconnect attempts use stale token → loop. Нет refresh
   pathway. Может не affect MVP (звонки < 2h), но edge case.

10. **`_remoteVideoTrack` always picks first remote participant's first camera
    publication** (line 453-457, 451). В group call doesn't pick «active speaker»
    — random first participant. Group is deferred но architecture не готова к этому.

11. **`Permission.microphone.request()` in `_requestPlatformMediaPermissions`
    auto-prompts** — `lib/services/call_coordinator_service.dart:1314`. Если user
    denied before, prompt не появится — silent return false. Нет UX path explain
    «вам нужно открыть настройки». `_openSystemSettings` exists в CallScreen но
    только когда `hasMediaPermissionIssue` — это reactive, не proactive.

## Recommended priority order

Список упорядочен по: **impact × user-visible severity × fix safety**.
Verified bugs (где код clearly broken) ставлю выше device-only suspicions
(где код выглядит correct но симптом наблюдается). Server-side изменения
запрещены (Артём explicit), значит fixes которые требуют backend endpoint —
flagged as «requires backend coordination».

1. **Add Android `FOREGROUND_SERVICE_MICROPHONE` + create CallForegroundService
   (Bug A + Bug 1)** — самый высокий impact, addresses production-breaking
   Bug #1. Fix complexity MEDIUM, no backend coupling, single PR scope. Safe
   rollout — без service, current code path still works on Android < 14;
   с service, audio capture survives backgrounding на всех version'ах.

2. **Fix camera state survival через `_refreshActiveCall` (Bug 2 + Bug 3)** —
   addresses Bugs #2 + #3 reported. Stop resetting `_cameraEnabled` в
   `_applyCall` если call.id unchanged. Add `_isTogglingCamera` guard.
   Requires careful test — touches hot path `_applyCall`. Code-only fix,
   medium safety. Coordinate с Bug 2 main fix: explicit `TrackPublishedEvent`
   handler.

3. **Add `BLUETOOTH_CONNECT` permission + runtime request (Bug B)** — silent
   BT failure для significant subset of users. Trivial fix, very safe.

4. **Fix PiP drag gesture conflict (Bug 4)** — addresses Bug #4 reported.
   Rewrite с `Listener` raw pointers или explicit `RawGestureDetector` для
   clean arena claim. Trivial-to-medium complexity. Verify on device.

5. **Verify mic publication post-`setMicrophoneEnabled` (Bug A pt 2 + Bug O)** —
   Check `audioTrackPublications` after await. Surface failure через UI.
   Trivial fix, complements item 1.

6. **Guard `_handleCoordinatorChanged` Navigator pop with callId match
   (Bug D)** — prevents collateral pop'ов в stacked scenarios. Trivial fix,
   very safe.

7. **Fix `_handleRoomDisconnected` reconnect loop с backoff (Bug G)** —
   prevents pathological battery drain edge case. Medium complexity. Add
   max retry + exponential backoff.

8. **Stop ringer race condition fix (Bug E + Bug N)** — Cancel в-flight
   ringer init если `_stopRinger` fired. Trivial.

9. **Pulse haptic on call connect + reactive UX polish (Telegram gaps)** —
   single-tap speaker toggle, swipe-to-accept gesture, vibration on connect.
   Trivial-to-medium, batchable in one PR.

10. **`toggleMicrophone` error handling (Bug O)** — wrap try/catch, surface
    via snackbar if mic toggle fails. Trivial.

11. **Filter coordinator notify events (Bug H)** — performance improvement,
    medium complexity. Skip rebuilds на stats events.

12. **Call history feature** — large gap vs Telegram. Requires backend
    endpoint coordination. Defer until backend window opens.

13. **iOS CallKit / PushKit (Bug C)** — large scope, native iOS work.
    Hard. Schedule when iOS prioritized (App Store rolled).

14. **Address remaining smaller items** in batches: Bug F, Bug I, Bug J,
    Bug K, Bug L, Bug M, Telegram polish items.

## ⚠️ Critical findings (requires immediate Артёмов attention)

**Bug A (no foreground microphone service)** — production-breaking. На Android 14+
(который > 50% Android install base в 2026), audio one-way симптом будет
наблюдаться у significant fraction of users в любой момент когда экран
блокируется или приложение backgrounded. Это не device-specific quirk — это
documented Android behavior change. **Без fix эта функция считается broken на
modern Android.** Recommend этот fix как Phase 2 first PR. Single-file
manifest change + ~80 lines Kotlin + ~50 lines Dart bridge = ~5-8 hours work.

**Bug 1 sub-cause #2 (silent mic publication failure)** — может маскировать
любой permission revocation, не только foreground service issue. Verify
`audioTrackPublications.any((p) => p.source == TrackSource.microphone)`
после `setMicrophoneEnabled` обязателен для proper error surfacing. **Без
этого debugging Bug 1 в future будет slow** — никаких сигналов в logs кроме
«нет звука».

## Iteration 5 — 2026-05-22T~18:30Z (direct dispatch, Q1 false-positive)

**Phase**: 2 (fix) — direct dispatch post Bug 2/3 device-verify
**Pick**: Q1 (commit `b914905`) false-positive race. Артём reported
после device test Bug A + Bug 2/3 что mic banner «микрофон не
подключился» fires хотя peer actually слышит.
**Action**:
* **Phase 1** root cause confirmed:
  * `_publishLocalMicrophone` (line 1404) использовал двойной check:
    `publication != null && participant.isMicrophoneEnabled()`
  * `isMicrophoneEnabled()` lookup race'нулась — между
    `setMicrophoneEnabled(true)` await return и tracks list update
    micro-window где lookup returned false → published=false →
    banner false positive.
  * Дополнительный bug: `_syncMediaStateFromLiveKit` (Fix C, line 1369)
    updates `_microphoneEnabled` / `_cameraEnabled` но никогда не
    clear'ил `_microphonePublishFailed`. Banner persisted даже после
    event-driven sync corrected mic to enabled.
* **Phase 2** decision: combined approach (minimal):
  1. Drop race-prone `isMicrophoneEnabled()` immediate check. Trust
     `publication != null` API contract.
  2. Extend `_syncMediaStateFromLiveKit` — defensive clear of
     `_microphonePublishFailed` когда LiveKit truth confirms mic
     published.
  * Rejected «remove Q1 entirely» — теряли immediate signal на real
    failures (Android 14 mic revoke когда foreground service absent —
    pre-Bug A scenario). Q1 backup remains valuable для other failure
    modes (codec init, HAL error).
  * Rejected «defer Q1 500ms» — adds latency на success path, всё
    равно race с device-specific timings.
* **Phase 3** implementation:
  * `_publishLocalMicrophone`: `published = publication != null`
    (single condition). Removed `isMicEnabled` debug log args.
  * `_syncMediaStateFromLiveKit`: добавлен conditional clear
    `if (micActual && _microphonePublishFailed) { _microphonePublishFailed = false; changed = true; }`
  * `debugApplyMediaSync` test seam — mirror production defensive
    reconcile.

    Checks `Permission.bluetoothConnect.status` — skip уже-granted
    либо permanently-denied/restricted (не спамим dialog). On undecided
    → `request()`. Swallows exceptions (best-effort).
  - Wired в `_requestPlatformMediaPermissions` через `unawaited(...)`
    после mic check, перед video-mode gate. Parallel'ный поток,

## Контекст по истории

В предыдущих сессиях (всё ещё в branch `claude/strange-pascal-3c3c1b`,
в main уже залито) были небольшие фиксы adjacent к этому
рефакторингу:

* Backend `linkPersonToUser` теперь использует `additive: true`
  при `applyCanonicalProfileToPerson` — не перезатирает данные
  существующих person-слотов когда новый user привязывается.
* Новый endpoint `DELETE /v1/trees/:treeId/persons/:personId/user-link`
  для отвязки user'а от слота.
* `ActiveChatTracker` + serverside chat-active gating — пуши
  не идут в активный чат.
* TreeMutationHistory — undo/redo для relations и person
  edit/delete на клиенте.

Это всё **не блокирует** рефакторинг и работает в parallel с
текущей моделью. Не ломай ничего из этого.

## Зачем именно этот промпт?

* Он **жёстко фиксирует scope** этой первой сессии (только
  audit, не код), чтобы агент не убежал делать всё сразу.
* Он указывает **конкретные deliverables** (3 файла) с
  чёткой структурой, чтобы можно было понять — done or not.
* Он перечисляет **forbidden actions** (не трогать код,
  не пушить в main), потому что для рефакторинга такого
  масштаба «сделал по-быстрому» = катастрофа.
* Он отсылает к PLAN.md как к источнику правды — и не
  дублирует его содержимое, чтобы не было drift'а между
  промптом и планом.

# Profile UX Redesign Proposal

> **Status**: design proposal awaiting Артёмов final sign-off перед implementation
> **Brainstorm**: Артём + Claude, 2026-05-28
> **Source evidence**: Артёмов скриншоты текущего профиля Лидии Александровны Мочалкиной + мама-friendly UX requirement + Telegraph/VK статьи style reference
> **Supersedes**: Phase D Pillar 1 (Profile redesign) original scope — расширяет к полному переписыванию профилей с нуля
> **Estimated timeline**: 4-6 weeks реализации
> **Risk profile**: high (touches central user surface, affects 70+ existing profiles, changes daily-use UX)

---

## 1. Context

### 1.1 Текущее состояние

Артёмов скриншот профиля Лидии Александровны Мочалкиной surface критическую UX проблему — **визуальная каша**. На одном экране одновременно:

- Hero с фото + именем + датами + бейджем «† Память» + «Для вас: Бабушка»
- Две кнопки сверху: «Это моя карточка» / «Приватность»
- Бейдж «Карточка в дереве»
- Обучающий текст «Пока это только карточка в дереве. Здесь лучше всего собирать семейные сведения, фотографии и память о человеке.»
- Секция «Связь»: «Бабушка · кровное родство»
- Секция «Путь»: «Кузнецов Артём → Кузнецова Наталья → Мочалкина Лидия»
- Секция «О человеке»: дата рождения, девичья фамилия
- «История изменений»: «Добавлено фото · 27 мая, 03:50 · 17 записей · Открыть историю»
- Секция «Связи и родство»: 3 кнопки (Добавить родственника / Путь родства / Исправить связи) + текст «Проверка работает поверх нового графа дерева...»
- Список семьи (муж, внук, сын, отец, мать, дочь)
- «Кому видна эта карточка?» — 3 радио-кнопки с длинными правилами
- Кнопка «Удалить из дерева» внизу

Один экран пытается быть просмотром + управлением приватностью + связями + историей правок + действиями + обучением. Шесть разных задач = **когнитивная перегрузка**.

### 1.2 Артёмов vision

Verbatim из brainstorm 2026-05-28:

> «Зачем пользователям (особенно маме) видеть "Это моя карточка", "Приватность". Зачем эта отдельная плашка "связь" - Уже же сверху написано, что это бабушка...»

> «И ниже с историй изменений тоже кажется так много непонятных для "Мамы" кнопок и текста...»

> «И убери режимы в редактировании (простой, расширенный) - практика показала, что это неудобно и непонятно!»

> «У нас должен быть самый удобный инструментарий для заполнения профилей! А сейчас это пока каша какая-то...»

> «Так что первым делом мы должны продумать верстак, а потом туда класть инструмент!»

> «У пользователей в приложении информация в "я" (наш личный кабинет) и в карточке инфа должна быть одинаковая.»

> «Я еще думаю над форматом как в Telegrapgh(статьи в тг)/vk статьи...»

### 1.3 Reference patterns

**Telegraph / telegra.ph:**
- Минималистичный full-screen editor
- Один большой пустой лист — placeholder, начинаешь печатать
- Toolbars прячутся, видны при focus / выделении
- «+» icon между параграфами для media insertion
- Auto-save
- Никаких modal pickers

**VK статьи:**
- Floating toolbar при выделении текста (bold/italic/header/link)
- «+» button в свободной зоне для media
- Простой режим без power-user complexity

**Википедия (по сути, не визуально):**
- Article-style биография
- Inline images mixed with text
- Multi-author collaborative editing
- Edit history
- Read-first, edit when needed

### 1.4 Strategic insight

Профиль = **центральное место в Rodnya**. Юзеры проводят там время сохраняя семейную историю. Если UX каша — приложение не сработает как «личный архив семьи».

Это **критическое foundation для Phase D** (Memory + History). Без чистого профиля Phase D pillars (interview prompts + voice input + biography sections) не приземлятся правильно.

## 2. Vision

### 2.1 Принципы

**1. Один экран — одна цель.** Просмотр карточки = смотрим на человека. Управление, история, удаление — в отдельных секциях по тапу.

**2. Mama-friendly UX как baseline.** Что на главном экране — должно быть понятно человеку 70+ лет с минимальным IT-опытом.

**3. Минимализм Telegraph-style + тёплый колорит.** Toolbars прячутся, никаких полей и табов на первом экране, серый фон не серый.

**4. Один режим редактирования.** Никаких «простой/расширенный».

**5. Свой и чужой профиль — одинаковая структура.** «Я» в приложении = просто карточка своей персоны.

**6. Биография — главный контент.** Article-style свободная форма с встроенными фото/голосом.

**7. Голос = акселератор текста, не interview tool.** Voice STT даёт текст быстрее, аудио сохраняется опционально.

**8. Multi-author collaborative editing.** Семья совместно пишет память. Attribution subtle (точки `·` в конце абзацев).

**9. Memorial = только data marker.** Никаких account effects, никаких logout. Безопасно от шутников.

**10. Существующие данные сохраняются.** Миграция через В: structured fields → меню «...», опциональный auto-generate черновик статьи.

### 2.2 Эстетика: тёплый/семейный

Не Telegraph-холодный минимализм. Атмосфера семейного альбома:

- **Шрифт**: serif для biography body (например, Lora, Merriweather, либо Source Serif Pro) — традиционный, тёплый. Sans-serif только для UI controls.
- **Цвета**: тёплые мягкие тона. Кремовый фон вместо чисто-белого. Photo frames с тонкой рамкой.
- **Spacing**: щедрые поля, не плотная вёрстка. Дышит.
- **Иконки**: rounded, soft, не sharp. Цвет — мягкие пастели.
- **Анимации**: плавные, не быстрые. Тёплые transitions.

Не grim memorial style. Не corporate clean. Просто **семейное** — как фотоальбом бабушки на полке, только цифровой.

## 4. Edit mode UX (Telegraph-inspired)

### 4.1 Открытие пустой биографии

```
←   Лидия Александровна Мочалкина          ✓ Готово
    Бабушка · 1949–2022
─────────────────────────────────────────────
                                                    
                                                    
    Расскажи о Лидии...                             
                                                    
                                                    
                                                    
                                                    
                                                    
                                                    
─────────────────────────────────────────────
   📷       🎤       💡
   фото    голос    идеи
```

- **Имя и отношение** — фиксированный заголовок наверху, не редактируется здесь (только в «...» → Основная информация)
- **Большой пустой лист** — placeholder «Расскажи о Лидии...»
- **Bottom toolbar** — 3 иконки минимально

### 4.2 «💡 идеи» открывает темы

Тап на «💡 идеи» → bottom sheet:

```
С чего начать?
─────────────────────────────────────
👶  Детство
   Где родилась? Какой был дом?

👨‍👩‍👧  Семья
   Родители, братья-сёстры

💑  Свадьба
   Как встретила супруга?

💼  Работа
   Чем занималась?

⚔️  Война
   Воспоминания о тех годах

🎨  Хобби
   Чем увлекалась?

✏️  Свой раздел...
```

(Темы — короче списка из Phase D Q2 ответа Артёма: «меньше 100 вопросов, только важнейшие»)

Тап «Детство» → промпт inserts вверху + раздел auto-creates:

```
←   Лидия Александровна Мочалкина          ✓ Готово
─────────────────────────────────────────────
                                                    
    💭 Где родилась бабушка? Какой это
       был город или село?                          
                                                    
    │  ← курсор, начинай печатать                   
                                                    
─────────────────────────────────────────────
   📷       🎤       💡
   фото    голос    идеи
```

Промпт-вопрос — лёгкая светло-серая подсказка. Юзер пишет ответ — промпт тускнеет. После сохранения — раздел называется «Детство».

### 4.3 Активное редактирование

Юзер пишет текст:

```
←   Лидия Александровна Мочалкина          ✓ Готово
─────────────────────────────────────────────
                                                    
    Лидия родилась 12 февраля 1949 года в           
    селе Иваново Калужской области. Их дом          
    стоял на краю деревни, рядом с лесом.│          
                                                    
─────────────────────────────────────────────
   📷       🎤       💡
```

**Никакого toolbar при простом наборе.** Маме не на что отвлекаться.

### 4.4 «+» между абзацами

Когда курсор на пустой строке между параграфами — маленькая `+` иконка на левом крае:

```
    Их дом стоял на краю деревни.                   
                                                    
    ⊕  ← маленькая «+» на пустой строке             
                                                    
    В детстве она помогала маме.                    
```

Тап `+` → quick menu:

```
    ⊕ +Раздел
    📷 Фото
    🎤 Голос
    🔗 Ссылка на родственника
```

### 4.5 Заголовки разделов

**Два способа:**

**Способ 1 (для мамы)**: «+Раздел» в quick menu выше. Явный путь.

**Способ 2 (для power user)**: выделил слово → floating toolbar → «Заголовок»:

```
       Свадьба
       ────────
       [Заголовок]  [B]  [I]
```

Тап «Заголовок» → текст становится разделом (большой шрифт, отступ).

### 4.6 Фото inline

Тап «📷» либо «+ Фото»:

```
[Открывается галерея устройства / камера]
   ↓
   Выбрал фото
   ↓

    Их дом стоял на краю деревни.                   
                                                    
    ┌──────────────────────┐                        
    │                       │                       
    │     [ФОТО ДОМА]       │                       
    │                       │                       
    └──────────────────────┘                        
    Подпись (необязательно)                         
                                                    
    В детстве она помогала маме.                    
```

- Full-width inline
- Подпись опциональна, серый плейсхолдер
- Tap на фото → меню: Заменить / Удалить / Подпись

### 4.7 Авто-галерея

Если автор вставляет 2+ фото подряд за ~1 минуту:

```
    ┌───┬───┬───┬───┐                                
    │ 1 │ 2 │ 3 │ 4 │   ← grid                       
    └───┴───┴───┴───┘                                
    Тап → fullscreen с свайпами                     
```

### 4.8 Голос как акселератор текста

Тап «🎤» в bottom toolbar:

```
─────────────────────────
│   ◉ Запись...           │
│                         │
│   ⏺  ⏸  ◼              │
│   00:23                 │
│   ▁▂▃▄▅▆▅▄▃▂▁           │
─────────────────────────
[Отмена]            [Готово]
```

После стопа — preview:

```
    ┌─────────────────────────┐                     
    │ ▶ ▁▂▃▄▅▆▅▄▃▂▁  00:23   │                     
    └─────────────────────────┘                     
                                                    
    📝 Транскрипт:                                  
    «Лидия родилась в селе Иваново,                 
    оно было небольшое, домов десять...»            
    [Скрыть текст]   [Исправить текст]              
                                                    
    [Перезаписать]  [Вставить в статью]             
```

**Транскрипт вставляется в курсор как обычный текст**, не как audio player.

Аудио файл сохраняется в backend но **по умолчанию свёрнут**. Под получившимся текстом маленькая ссылка:

```
    Лидия родилась в селе Иваново,                  
    оно было небольшое, домов десять.               
    Зимой все собирались у печки...                 
    🎵 Прослушать оригинальную запись               
```

Тап на ссылку → audio player разворачивается inline. Если не нужен — никто не знает что аудио было.

**Audio file всегда сохраняется** — внуки через 50 лет могут переслушать бабушкин голос. Артефакт preserved.

### 4.9 Mentions (ссылки на родственников)

Тап «🔗» в quick menu → choice:

```
🔗 На кого ссылаемся?
─────────────────────────
○ На родственника
○ На внешний сайт
```

«На родственника» → picker среди членов семьи:

```
Выбери родственника:
─────────────────────────
👤 Мочалкин Геннадий — муж
👤 Кузнецов Артём — внук
👤 Мочалкин Евгений — сын
... [scroll]
```

Тап → в тексте появляется clickable mention:

```
    ...вместе с [мужем Геннадием] они переехали...
                  ^^^^^^^^^^^^^^^^^                  
                  clickable link к карточке Геннадия 
```

Mentions auto-sync — если Геннадия переименовали в основной информации, link text updates везде.

### 4.10 Авто-сохранение

```
←   Лидия Александровна Мочалкина          ✓ Готово
    ✓ Сохранено · 3 сек назад
```

- Каждые 5 секунд либо при потере фокуса
- Маленький индикатор внизу

### 4.11 Выход из редактора

- Кнопка «✓ Готово» вверху справа → финальный auto-save + переход в режим просмотра
- Кнопка «←» вверху слева → возврат назад (тоже auto-save)

## 5. Multi-author collaborative editing

### 5.1 Принцип

Любой member семьи может писать в карточку любого родственника:

- Добавлять новые абзацы
- Редактировать существующие (с сохранением истории)
- Добавлять/удалять фото/аудио

### 5.2 Attribution

**Visible by default**: ничего. Чтение чистое.

**Subtle marker**: точка `·` в конце последнего слова каждого абзаца, серая, невидимая при беглом чтении.

```
    Лидия встретила деда в 1968 году. Они познакомились на танцах
    в доме культуры села Иваново·
```

**Tap на `·`** → mini-card popup:

```
┌─────────────────────────┐
│ Записал Артём Кузнецов  │
│ 25 мая 2026, 03:42      │
│ [Открыть профиль автора] │
└─────────────────────────┘
```

**Внизу каждой главки**: список авторов (без дат):

```
    После свадьбы они переехали в районный центр·
    
    Соавторы раздела: Артём, Наталья, Евгений
```

**Полная история** доступна через меню «...» → «История изменений» (см. 3.2.4).

### 5.3 Конфликты

Если два автора редактируют один абзац одновременно (rare случай):

- Last-write-wins на уровне абзаца
- Старая версия сохраняется в audit log
- В «История изменений» — обе версии visible с rollback option
- Notification к обоим: «Артём и Наталья редактировали один абзац. Посмотри историю.»

Не явный «merge UI» — слишком сложно для мамы. История сохраняет всё, rollback доступен.

### 5.4 Кто может редактировать

Per Phase B membership model:

- **Owner / Editor семьи** — полный доступ к карточкам родственников в семье
- **Viewer семьи** — только просмотр
- **Hidden filter** (Phase B FE7) — если кто-то скрыл персону у себя, она невидима но другие могут редактировать
- **Если человек живой и зарегистрирован** — full ownership своей карточки, может удалить любые чужие добавки

## 6. Self profile (мой профиль)

### 6.1 Идентичная структура

Карточка «Я» (свой профиль) = такая же структура как чужие. Никаких отличий в визуальной архитектуре.

**Hero:**
```
              [круглое фото]
              
        Артём Кузнецов
        25 лет

     ┌──────────────────────────────────────┐
     │  ✏️  Рассказать о себе               │
     └──────────────────────────────────────┘
```

Без CTA «Спросить» / «Послать вопрос» — это для родственников.

### 6.2 Идеи для себя

«💡 идеи» открывает темы адаптированные:

```
С чего начать?
─────────────────────────
👶 Где я родился
🎓 Где учился
💼 Чем занимаюсь
🎨 Что люблю
👨‍👩‍👧 Моя семья сейчас
✏️ Свой раздел...
```

### 6.3 Привязка к личному кабинету

Существующий «Я» (settings/profile screen) **rewrites к карточке-формату**:

- Settings (уведомления, приватность, аккаунт) — переезжают в отдельную ✓
  «Настройки» секцию приложения, не в карточку
- Карточка «Я» = только биография (как у родственников)
- Логаут, удаление аккаунта, etc. — в Настройки приложения

Единая модель: моя карточка и карточка бабушки **одинаково устроены**, только promo CTAs отличаются (на своей карточке нет «спроси»).

## 7. Living vs Deceased differentiation

### 7.1 Memorial = только data marker

**Никаких account effects.** Если родственник был с аккаунтом в Rodnya:
- Marker «† Память» на карточке
- НИКАКОГО logout
- НИКАКОГО ban от login
- Аккаунт продолжает работать (тeoreтически)

Это закрывает vulnerability (шутник не может разлогинить семью через memorial mark).

### 7.2 Как ставится memorial

```
Меню «...» → «Основная информация» → редактирование →
Поле «Дата смерти» → ввод даты → ✓ Сохранить
```

Marker появляется автоматически когда заполнена «Дата смерти».

### 7.3 Что меняется визуально

| Элемент | Живой | Ушедший |
|---|---|---|
| Hero label | «Бабушка · 76 лет» | «† Память: Бабушка · 1949 — 2022» |
| Primary CTA | «✏️ Добавить историю» + «💬 Послать вопрос» | «✏️ Добавить воспоминание» |
| Promпт-идеи | «Спроси: где родилась?» | «Расскажи: что помнишь про её еду?» |
| Tone | Активный | Memorial без grim — тёплый |

### 7.4 Phase D «Послать вопрос» только для живых

Phase D pillar 2 (interview prompts) — отправка вопроса родственнику только если он жив и зарегистрирован в приложении.

- Кнопка «💬 Послать вопрос» visible только на карточке живого + зарегистрированного
- Если не зарегистрирован — кнопка отсутствует
- Если ушёл — кнопка отсутствует

## 8. Visual identity

### 8.1 Тёплый/семейный стиль

#### Цветовая палитра

```
Фон:          #FAF7F2 (тёплый кремовый, не белый)
Карточки:     #FFFFFF (чисто-белые, чтобы выделялись)
Текст:        #2C2C2C (мягко-чёрный, не #000)
Вторичный:    #6B6B6B (серый для метаданных)
Акцент:       #5B8B3E (приглушённый зелёный, как наш текущий primary)
Memorial:     #8B7B5A (тёплый бежевый для «†» badge — не серый/чёрный)
```

#### Типографика

- **Headers (раздeлы статьи)**: serif (Lora либо Source Serif Pro), 24px, semi-bold
- **Body text**: serif (тот же шрифт), 17px, normal weight, line-height 1.6
- **UI controls** (кнопки, метки): sans-serif (нaш текущий шрифт системы)
- **Метаданные** (Соавторы, даты): sans-serif, 13px, серый

Serif для чтения — теплее ощущается, ближе к книге / альбому. Sans-serif для UI — функционально читаемо.

#### Spacing

- Щедрые поля: 24px по краям
- Между параграфами: 16px
- Между разделами: 32px
- Inline media — full-width с 16px вертикальным spacing

#### Photo styling

- Inline фото — full-width, rounded corners 12px, subtle shadow
- Photo frames с тонкой кремовой рамкой (как физический альбом)
- Avatar в hero — круг 120px, subtle border

#### Анимации

- Все transitions ≥ 200ms (плавно, не fast)
- Спокойные fade-in для фото при scroll
- Никаких bouncy либо elastic анимаций

### 8.2 Иконография

Rounded, soft style. Material Symbols Rounded либо аналог.

В bottom toolbar:
- 📷 → camera_rounded
- 🎤 → mic_rounded  
- 💡 → lightbulb_rounded

## 9. Migration plan

### 9.1 Existing data preserved

**Все existing данные сохраняются**, ничего не теряется:

| Existing | Новое местоположение |
|---|---|
| Имя, фамилия, девичья | Меню «...» → Основная информация |
| Даты рождения / смерти | Меню «...» → Основная информация |
| Отношение | Меню «...» → Основная информация |
| Фото (avatar + gallery) | Avatar → hero, остальные → Меню «...» → Все фото |
| Visibility settings | Меню «...» → Кто видит карточку |
| Audit log (история правок) | Меню «...» → История изменений |

### 9.2 Биография при первом open

При открытии профиля с пустой биографией:

```
←   Лидия Александровна Мочалкина          ⋯ ✏️
─────────────────────────────────────────────
              [фото]
        Бабушка · 1949 — 2022
─────────────────────────────────────────────

    Биография ещё не написана.

    ┌─────────────────────────────────────┐
    │ Можно создать черновик из основных  │
    │ данных — потом отредактируешь.       │
    │                                     │
    │  [Создать черновик]  [Я сам напишу] │
    └─────────────────────────────────────┘

─────────────────────────────────────────────
   📷       🎤       💡  (если в edit mode)
```

#### «Создать черновик» — auto-generated stub

Из existing structured fields:

```
## Основное

Лидия Александровна Мочалкина (девичья фамилия 
Супрунова) родилась 12 февраля 1949 года.

[фото из avatar — inline]

## Прощание

Ушла 14 декабря 2022 года.
```

Авто-генерация simple — facts из data в плавный русский текст. Стиль нейтральный, юзер сам перепишет к более тёплому.

#### «Я сам напишу» — empty article

Просто открывает редактор с placeholder «Расскажи о Лидии...» и кнопкой «💡 идеи» в bottom toolbar.

### 9.3 Не делается автоматически

- **Существующие фото в article не вставляются.** Они в галерее (меню «...» → Все фото). Юзер может вручную добавить нужные в article при редактировании.
- **Audit log не мигрируется в новую систему.** Старый log preserved в меню «...» → История изменений, новые правки добавляются туда же.

### 9.4 Phase B/C compatibility

- Phase B федеративные семьи compatible: каждая карточка bound к семя, multi-author = семья members
- Phase C круг расширение compatible: друзья tier работает same way
- Phase D voice + prompts integration: voice = accelerator (надиктовать), prompts = «идеи» в редакторе

## 12. Risk assessment

| Риск | Severity | Mitigation |
|---|---|---|
| Существующие 70+ профилей теряют данные при миграции | 🔴 Critical | Migration script обратимый, structured data сохраняется в «...», article body новый | 
| Editor сложен для мамы (Telegraph minimalism может быть пугающим) | 🟠 Major | Промпт-идеи + auto-generated stub снимают «белый лист» fear |
| Multi-author conflicts визуально неприятные | 🟠 Major | Attribution subtle (точки), история в меню, last-write-wins per paragraph |
| Voice STT на устройстве плохо распознаёт русский | 🟠 Major | Юзер видит транскрипт, может редактировать. Аудио всегда preserved. |
| Privacy settings скрыты — юзеры не находят | 🟡 Medium | Onboarding гайд при первом создании карточки, tooltip в меню |
| Migration breaks Phase B / Phase 4 extended network | 🟠 Major | Карточки backwards compat — backend endpoints stable, только UI меняется |
| Tree visualization integration ломается | 🟡 Medium | «Открыть в дереве» в меню preserves existing tree feature |
| Performance — длинные article рендерятся медленно | 🟡 Medium | Lazy load для media, pagination для длинных статей если >50 параграфов |
| Memorial marker abuse (хотя мы убрали logout) | 🟢 Low | Только data marker, любой member может убрать |

## 14. Open questions для финального approve

Это последний шаг перед implementation:

### Q1 — Migration auto-generated stub: opt-in либо opt-out?

- **Opt-in** (мой proposal): юзер видит выбор «Создать черновик / Я сам напишу»
- **Opt-out**: stub генерируется автоматически, юзер может «начать заново»
- **No stub**: пустая статья всегда, только промпты

**Recommend**: opt-in (мой proposal). Юзер контролирует.

### Q2 — Memorial date — кто может проставлять?

- **Любой member семьи** — flexible
- **Только owner карточки** — restrictive
- **Sam person (если живой и зарегистрирован)** — self-control

**Recommend**: любой member семьи. Marker = data only, без destructive effects.

### Q3 — Multi-author last-write-wins per paragraph: достаточно?

- Конфликты редки в семье (5-10 родственников)
- Если случается — история сохраняет всё
- Notification к обоим авторам когда конфликт

**Recommend**: да, достаточно. CRDT либо OT overkill для семейного приложения.

### Q4 — Самый первый user flow когда мама открывает свежую установку

- Дойдёт ли она до «Заполнить информацию»?
- Нужен ли onboarding tutorial первый раз?
- Либо просто empty card с promпт-идеями?

**Recommend**: 30-секундный onboarding overlay при первом открытии карточки — «Вот тут карточка. Нажми ✏️ чтобы писать. Не бойся ничего, всё сохраняется автоматически.»

### Q5 — Serif font выбор — Lora vs Source Serif Pro vs другой?

- **Lora**: бесплатный, тёплый, русский поддержка
- **Source Serif Pro**: Adobe, более formal
- **PT Serif**: русский народный, разработан в Paratype
- **Noto Serif**: Google, нейтральный

**Recommend**: PT Serif (Paratype) — русский-native шрифт, тёплый, отлично кириллица.

### Q6 — «Идеи» промпты вынести в multi-tier (sub-prompts)?

Например:
- «Детство»
  - «Где родилась?»
  - «Какой был дом?»
  - «Любимые игры?»

**Recommend**: на этом этапе — одноуровневые промпты («Детство»). Sub-prompts добавятся в Phase D+1.

### Q7 — Voice recording max длина?

- **Unlimited** — generous
- **5 минут** — typical interview
- **15 минут** — comprehensive

**Recommend**: 15 минут максимум на запись. Если нужно дольше — несколько записей подряд (smart concat).

### Q8 — Photo gallery в меню «...» — sortable?

- По дате (newest first)
- По разделу статьи (где использовалось)
- Custom user order

**Recommend**: по дате (newest first). Custom order Phase D+1 если будет request.

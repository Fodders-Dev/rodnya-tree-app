# План доработки дерева

Дата: 18 апреля 2026

## Что уже закрыто в этом проходе

- Карточка родственника теперь использует graph snapshot глубже, а не только короткий viewer label.
- В `FamilyRelation` подняты метаданные `parentSetId`, `parentSetType`, `isPrimaryParentSet`, `unionId`, `unionType`, `unionStatus`.
- На экране деталей появился нормальный `Путь родства`:
  - основной label
  - кровная / некровная связь
  - число альтернативных путей
  - path summary
  - пошаговая цепочка людей и прямых связей
- На экране деталей появился лист `Другие родители` для нескольких parent sets.
- Прямые связи в relation management теперь показывают контекст: тип parent set и статус/тип союза.

## Следующая волна: надежность графа

### 1. Валидация графа перед сохранением

Цель: не только красиво рисовать дерево после ошибки, а предотвращать появление плохих состояний.

Нужно добавить серверные проверки:

- ребенок не должен оказываться одновременно в двух primary parent sets одного типа без явного подтверждения
- union не должен визуально или семантически превращаться в sibling row
- человек не должен переезжать в чужую ось поколения из-за неполного parent set
- при удалении или смене связи дерево должно проверять, не осталось ли `висячих` детей без family unit
- graph snapshot должен возвращать warnings для UI, если найдены:
  - конфликт поколений
  - несколько конкурирующих primary parent sets
  - узел без связей внутри branch
  - циклические или амбивалентные direct links

UI-поведение:

- показывать короткий warning chip только когда реально найден конфликт
- в relation editor давать не просто `Удалить связь`, а объяснять, что именно сломается или исправится

### 2. Автоисправление типовых ошибок

Нужно добить normalization rules:

- `add partner` всегда создает отдельный union и никогда не влипает в sibling rail
- `add parent` сначала пытается дополнить текущий parent set нужного типа
- `add sibling` наследует полный primary parent set якоря
- `change relation type` должен уметь безопасно переносить relation между `biological`, `guardian`, `step`, `adoptive`

## Следующая волна: импорт из старого плаката

По приложенному дереву видно, что архивная схема очень плотная. Автоматически строить из нее живое дерево без подтверждения опасно: на плакате много близко стоящих рамок, пересечений ветвей и повторяющихся фамилий.

Поэтому нужен не `магический импорт`, а подтверждаемый мастер:

### 3. Draft import assistant

- пользователь загружает изображение старого дерева
- система выделяет карточки-кандидаты и пытается распознать:
  - имя
  - годы
  - положение в ветке
  - возможные пары
- каждому кандидату предлагается одно из действий:
  - привязать к существующему человеку
  - создать нового человека
  - пропустить
- все изменения сначала складываются в draft graph, а не в live tree
- только после подтверждения применяются к дереву

### 4. Matching engine

Нужны эвристики сопоставления:

- имя + фамилия + примерная дата
- близость к уже подтвержденным родственникам
- принадлежность к одной ветке
- фото-совпадение только как слабый сигнал, не как источник истины

### 5. Safe scope для первого релиза импорта

Первую версию ограничить:

- только ближайшие 2-3 поколения вокруг текущего пользователя
- без автоматического проведения сложных браков
- без автосоздания custom relations
- без автосклейки людей при низкой уверенности

## Следующая волна: словарь родства и кастомные связи

Автоматически должны выводиться только связи, которые реально считаются из графа крови и брака.

Отдельным слоем нужно оформить explicit relations:

- `кум`, `кума`
- `побратим`
- `примак`
- другие редкие социальные связи

Принцип:

- это отдельные typed custom edges
- они не участвуют в blood graph
- они не ломают поколения
- они могут отображаться в карточке и в relation path как боковая социальная связь

## Следующая волна: layout и визуальная шлифовка

### 6. Конфликт-роутинг линий

- меньше пересечений между grandparent blocks и parent row
- более жесткая привязка child bar к своему family unit midpoint
- защита от наложения союзов и детей при плотных поколениях

### 7. Стабильные стороны семьи

- paternal и maternal family units не должны «прыгать» местами при частичном изменении данных
- side assignment нужно считать от семейного блока, а не от текущего набора координат

### 8. Минимализм без потери смысла

- на canvas оставлять только главные chips
- все сложные пояснения уводить в bottom sheets
- легенда для dashed lines / secondary parent sets / past unions открывается по запросу

## Практический порядок выполнения

### Волна A

- server warnings для graph conflicts
- UI warnings в relation editor
- normalization для change relation type

### Волна B

- draft import assistant для архивного плаката
- candidate matching
- confirm/apply flow

### Волна C

- typed custom relations
- отображение social links в карточке и path sheet

### Волна D

- финальный layout polish
- production smoke на реальном дереве
- выкладка после ручной проверки сложных кейсов

## Риск, который надо держать в голове

Главная опасность не в том, что дерево нарисуется некрасиво, а в том, что UI даст пользователю создать структурно ложную семью. Поэтому следующие изменения должны идти в таком порядке:

1. сначала запрет/предупреждение плохих состояний
2. потом draft-import с подтверждением
3. только потом более агрессивный автоматизм

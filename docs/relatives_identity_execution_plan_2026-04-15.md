# План: родственники, события, identity

## Цель
- Вернуть на Home живые семейные события, а не только календарь праздников.
- Переделать фон приложения с нуля под более явный лесной/родовой мотив.
- Пересобрать редактирование родственника в два режима: быстрое и расширенное.
- Добавить фото/видео прямо в форму родственника.
- Зафиксировать модель дальнейшей склейки `офлайн карточка -> реальный пользователь` и работы одного человека в нескольких деревьях.

## Волна 1. Что делаем сейчас

### 1. Home events
- Поднять приоритет семейных событий над общим календарём.
- Увеличить лимит на главной, чтобы не терять дни рождения и семейные даты.
- Сохранить компактный rail и фильтры, но гарантировать, что если в дереве есть `Родня/Семья/Память/Повод`, они попадают на первый экран.

### 2. Новый фон
- Убрать текущие случайные ветки.
- Сделать новый backdrop с нуля: мягкий светлый лесной орнамент, берёзы, кроны, дальний лес, без тяжёлой иллюстрации.
- Держать фон тихим, чтобы не спорил с контентом.

### 3. Relative editor
- Ввести режимы `Основное` и `Расширенно`.
- В `Основное` оставить только быстрые факты:
  - имя
  - фамилия
  - отчество
  - пол
  - связь
  - дата рождения
- В `Расширенно` добавить:
  - девичью фамилию
  - дату смерти
  - место рождения
  - биографию
  - заметки
  - образование
  - кастомные важные события
  - дату свадьбы для spouse-связей

### 4. Media в форме
- Для нового родственника:
  - собирать черновые фото/видео до сохранения карточки
  - после создания человека догружать их в gallery
- Для редактируемого родственника:
  - дать быстрые кнопки `Фото` и `Видео`
  - показывать текущую галерею и черновики
  - не уводить пользователя в отдельный экран для базового media-pass

## Волна 2. Следующий слой после этого pass

### 5. Wikipedia-style editing
- Разделить карточку человека на:
  - краткий факт-блок
  - расширенные сведения
  - события
  - медиа
  - история изменений
- Хранить edit history как уже сделанный tree history, но довести UI до понятного “кто и что поменял”.

### 6. Offline profile -> real user ownership
- При приглашении офлайн-карточки в приложение:
  - если карточка ещё не занята, link `person.userId`
  - подтянуть пользователя в дерево как владельца своей карточки
  - не терять существующую историю изменений и media
- Если пользователь уже есть в другом дереве:
  - не считать это новым человеком “с нуля”
  - привязывать tree-specific person record к одной user identity

### 7. Один человек в разных деревьях
- Ввести явное разделение:
  - `User identity`: реальный человек/аккаунт
  - `Tree person record`: представление человека внутри конкретного дерева
- Не делать автоматический глобальный merge всех person records между деревьями.
- Делать безопасную модель:
  - один `userId`
  - несколько tree-specific cards
  - связь между ними через общий identity layer
- Отдельно понадобятся:
  - canonical person link
  - merge/suggest duplicates
  - conflict policy для имени, фото, дат и заметок

## Критерий готовности текущего pass
- На Home в первых событиях видны семейные даты, если они вообще есть в дереве.
- Фон действительно новый и читается как часть бренда, а не как случайный орнамент.
- В форме родственника есть явный переключатель `Основное / Расширенно`.
- Можно прикрепить фото и видео без обходного пути через details screen.
- Существующие create/edit flows и тесты не ломаются.
